## 1. Fix Critico: Cancellazione File dalla Media Gallery

### Sintomo

Premere il pulsante "Elimina" su qualsiasi file nella Media Gallery admin (ZIP, PDF, immagini) non produceva alcun effetto. Il file rimaneva nell'elenco e sul server senza messaggi di errore visibili all'utente.

### Causa Radice

Il metodo `deleteMedia` in `src/api.ts` inviava la richiesta DELETE senza parametri nell'URL:

```typescript
// PRIMA (errato)
const res = await fetch(`${API_URL}/media.php`, {
    ...fetchConfig,
    method: 'DELETE',
    body: JSON.stringify({ id })  // ID nel body, mai letto da $_GET
});
```

Il backend `public/api/media.php` leggeva l'ID esclusivamente dal query string (`$_GET['id']`), non dal body della richiesta. Le richieste HTTP DELETE non hanno un body standard leggibile via `$_GET`, quindi l'ID arrivava come `null` e il backend rispondeva con HTTP 400 ("ID mancante"). Il frontend riceveva l'errore ma il blocco `catch` in `MediaGallery.tsx` mostrava solo un `alert('Errore eliminazione file.')` generico, spesso soppresso dai browser moderni se la pagina non è in focus.

Il pattern corretto era già in uso in `deleteArticle` e `deleteProject`, che passano l'ID nel query string.

### Fix Applicato

**File modificato: `src/api.ts`**

```typescript
// DOPO (corretto)
deleteMedia: async (id: number) => {
    const res = await fetch(`${API_URL}/media.php?id=${id}`, {
        ...fetchConfig,
        method: 'DELETE',
    });
    if (!res.ok) throw new Error('Errore cancellazione media');
    return res.json();
},
```

Rimosso il `body: JSON.stringify({ id })` ridondante, aggiunto `?id=${id}` all'URL. Il backend `media.php` non richiede alcuna modifica.

### Impatto

La cancellazione dei singoli file funziona correttamente per tutti i tipi (immagini, PDF, ZIP, MP3). Il bug era presente dalla v1.3.4 (introduzione Media Gallery) ma non era stato rilevato perché il messaggio di errore era silenzioso.

## 2. Ottimizzazione Automatica Immagini WebP all'Upload

### Problema

Come rilevato nell'audit prestazioni del 30/03/2026 (documentato in roadmap, sezione 3 "Priorità Alta"), le immagini PNG generate da AI venivano caricate senza alcuna ottimizzazione (file da 5.6 MB, 2.7 MB, 2.6 MB), portando a sessioni di navigazione da 12-15 MB totali. La causa era strutturale: `upload.php` spostava i file sul disco senza alcuna fase di processamento.

### Soluzione Implementata

**File modificato: `public/api/upload.php`**

Aggiunto un blocco di ottimizzazione automatica che si attiva dopo il `move_uploaded_file()` iniziale, prima dell'inserimento nel database. La logica si attiva solo per immagini JPEG e PNG (non per WebP già ottimizzati, GIF, PDF, ZIP, MP3).

#### Flusso di ottimizzazione

1. **Rilevamento tipo**: controlla che `$realMime` sia `image/jpeg`, `image/png` o `image/gif`, e che la GD Library sia disponibile (`extension_loaded('gd') && function_exists('imagewebp')`)
2. **Caricamento immagine**: usa `imagecreatefromjpeg()`, `imagecreatefrompng()` o `imagecreatefromgif()` in base al MIME type
3. **Resize condizionale**: se la larghezza originale supera 1920px, ridimensiona mantenendo le proporzioni con `imagecopyresampled()`. Preserva la trasparenza PNG tramite `imagealphablending(false)` e `imagesavealpha(true)`
4. **Conversione WebP**: salva con `imagewebp($img, $webpDestination, 82)` — qualità 82, ottimo compromesso peso/qualità
5. **Pulizia**: elimina il file originale PNG/JPEG dal disco con `unlink($destination)`
6. **Aggiornamento variabili**: `$newFileName`, `$publicUrl` e `$fileName` (nome display) vengono aggiornati con estensione `.webp` prima dell'`INSERT` nel database
7. **Fallback silenzioso**: se GD non è disponibile o `imagewebp()` fallisce, il file originale rimane intatto e viene registrato nel DB com'era — nessun errore esposto all'utente

#### Note tecniche

- **GIF animate**: perdono l'animazione (solo primo frame convertito). Edge case accettato — le GIF non vengono usate come asset del sito
- **File già WebP**: saltati automaticamente dalla logica (MIME `image/webp` non è in `$imageMimes`)
- **Nessuna migrazione DB**: il campo `filename` nel database memorizza il nome originale con estensione `.webp`, coerente con il comportamento pre-esistente
- **Fallback hosting**: se la GD Library non è disponibile sull'hosting, l'upload avviene normalmente senza ottimizzazione. Verificare con `phpinfo()` o `extension_loaded('gd')`

#### Risparmio atteso per tipo

| Formato origine | Dimensione tipica | Dopo WebP q82 | Risparmio stimato |
|:---|:---|:---|:---|
| PNG AI-generated (4K) | 5-6 MB | ~0.8-1.2 MB | ~80% |
| PNG screenshot | 1-2 MB | ~0.3-0.5 MB | ~70% |
| JPEG fotografia | 2-4 MB | ~0.5-1 MB | ~65% |

## 3. Script Batch per Immagini Già Caricate

### File creato: `scripts/optimize_uploads.php`

Script PHP one-shot da caricare temporaneamente in `public/api/` per ottimizzare tutte le immagini JPEG e PNG già presenti nella cartella `/uploads/` prima di questa release.

#### Funzionamento

1. Verifica che la GD Library sia disponibile — termina con errore leggibile se assente
2. Scansiona l'intera directory `public/uploads/`
3. Per ogni file JPEG o PNG trovato:
   - Carica l'immagine con GD
   - Ridimensiona se la larghezza supera 1920px, mantenendo le proporzioni
   - Converte in WebP con qualità 82
   - Salva il nuovo file `.webp` nella stessa directory
   - Cerca il record corrispondente nel DB tramite `file_path` (`/uploads/hashname.ext`)
   - Aggiorna `filename`, `file_path`, `mime_type` e `size` nel DB
   - Elimina il file originale PNG/JPEG
4. Salta automaticamente: file già `.webp`, `.gif`, `.svg`, `.pdf`, `.zip`, `.mp3` e qualsiasi non-immagine
5. Produce un report testuale dettagliato per ogni file con: dimensione prima/dopo, risparmio percentuale, esito DB, note resize
6. Produce un riepilogo finale con totali MB prima/dopo e risparmio complessivo

#### Sicurezza

Segue il pattern sicurezza standard del progetto: **deve essere eliminato dal server immediatamente dopo l'esecuzione**. Non va mai lasciato in produzione in modo permanente. Stessa procedura di `emergency_pwd.php` e `fix_db.php` (già eliminati in versioni precedenti).

#### Utilizzo

```
1. FTP: caricare scripts/optimize_uploads.php in public/api/
2. Browser: visitare https://simonepizzi.it/api/optimize_uploads.php
3. Leggere il report nel browser (testo plain)
4. FTP: eliminare immediatamente il file dal server
```

## 4. Multi-Selezione e Bulk Delete nella Media Gallery

### Problema

La Media Gallery permetteva solo l'eliminazione di un file alla volta tramite il pulsante icona Cestino nella card. Con l'accumulo di file nel tempo, eliminare batch di file obsoleti richiedeva molti click ripetuti.

### Funzionalità Aggiunta

**File modificato: `src/pages/admin/MediaGallery.tsx`**

#### Stato React aggiunto

- `selectedIds: Set<number>` — set degli ID dei file attualmente selezionati
- `bulkDeleting: boolean` — true durante l'eliminazione bulk, disabilita il pulsante e mostra spinner

#### Selezione singola (checkbox per card)

Ogni card della gallery mostra una **checkbox** (icona `Square`/`CheckSquare` da Lucide React) nell'angolo in alto a sinistra:
- In stato normale: invisibile (`opacity-0`), appare on-hover tramite `group-hover:opacity-100`
- Se il file è selezionato: sempre visibile, colore `dis-green`
- La card selezionata riceve bordo `border-dis-green/50` e `ring-1 ring-dis-green/30` come feedback visivo
- Il click sulla checkbox non interferisce con gli altri controlli hover (Copia Link, Apri/Scarica)

#### Seleziona tutti / Deseleziona tutti

Nella tab bar, a sinistra del pulsante Refresh, appare il pulsante **"Seleziona tutti"** visibile solo quando ci sono file nella tab attiva:
- Al primo click: seleziona tutti i file della tab corrente
- Se sono già tutti selezionati: fa da toggle e li deseleziona tutti
- La selezione viene **svuotata automaticamente** al cambio tab (via `useEffect` su `activeTab`), evitando selezioni fantasma di file non visibili

#### Barra azioni contestuale

Quando almeno un file è selezionato, appare una **barra contestuale** tra l'header upload e la gallery che mostra:
- Contatore aggiornato in tempo reale: "N file selezionato" / "N file selezionati" (forma singolare/plurale corretta)
- Pulsante "Deseleziona tutti" con icona X
- Pulsante **"Elimina selezionati (N)"** in stile rosso con icona Trash2

#### Logica eliminazione bulk

```
1. window.confirm() con conteggio preciso
2. Loop sequenziale su tutti gli ID selezionati
3. api.deleteMedia(id) per ciascuno — errori singoli silenziati
4. Aggiornamento atomico di mediaList e selectedIds al termine
5. Messaggio di successo con conteggio reale file eliminati
```

Il loop sequenziale (anziché `Promise.all`) evita di sovraccaricare il server con DELETE simultanee, particolarmente rilevante per hosting condivisi con limiti di connessioni.
