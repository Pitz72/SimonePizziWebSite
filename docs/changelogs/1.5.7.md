# Changelog v1.5.7 - Security Fix & Ordinamento Articoli Programmati

**Data Rilascio:** 23 Marzo 2026

Questa release risolve due bug indipendenti emersi dall'analisi di sicurezza della codebase: una vulnerabilita di accesso non autenticato alle bozze tramite API e un difetto logico nell'ordinamento degli articoli programmati che ne alterava la posizione nel feed al momento della pubblicazione.

---

## 1. Fix Sicurezza: Accesso Non Autenticato a Bozze via API (?id=X)

### Problema

L'endpoint GET `/api/articles.php?id=X` non richiedeva autenticazione. Chiunque conoscesse (o indovinasse per enumerazione sequenziale) l'ID numerico di un articolo poteva chiamare direttamente l'API e ricevere il payload JSON completo (`SELECT *`) dell'articolo, **incluse bozze e articoli programmati per il futuro**, senza essere loggato nel pannello admin.

A differenza dell'endpoint gemello `?slug=X` (che implementava correttamente il controllo di visibilita pubblica con fallback a 404 per contenuti non pubblicati), il percorso `?id=X` -- usato esclusivamente dal React Editor del pannello admin per caricare un articolo in modifica -- era privo di qualsiasi guardia di autenticazione.

**Scenario di attacco:**
```
GET /api/articles.php?id=1  -> 200 OK (bozza esposta)
GET /api/articles.php?id=2  -> 200 OK (articolo futuro esposto)
GET /api/articles.php?id=3  -> 200 OK (contenuto riservato esposto)
```

### Causa Radice

Il branch `?id=X` nel blocco GET di `articles.php` era stato progettato come endpoint interno del pannello admin, assumendo implicitamente che solo l'admin vi accedesse. Mancava la chiamata esplicita ad `Auth::check()` che invece tutti gli altri metodi write (POST, PUT, DELETE, PATCH) implementavano correttamente.

Questo bug era presente dalla v1.3.1 (introduzione dell'Admin Editor React).

### Risoluzione

Aggiunta la chiamata `Auth::check()` come prima istruzione del branch `?id=X`, immediatamente dopo l'ingresso nel blocco condizionale. Se la sessione admin non e' presente, il server risponde con `401 Unauthorized` e termina l'esecuzione prima di interrogare il database.

**File modificato:** `public/api/articles.php` (riga 71)

```php
// PRIMA (vulnerabile)
if (isset($_GET['id'])) {
    $stmt = $pdo->prepare("SELECT * FROM articles WHERE id = ?");
    ...
}

// DOPO (protetto)
if (isset($_GET['id'])) {
    Auth::check();  // <-- aggiunto
    $stmt = $pdo->prepare("SELECT * FROM articles WHERE id = ?");
    ...
}
```

## 2. Fix Logico: Ordinamento Articoli Programmati nel Feed

### Problema

Gli articoli pubblicati tramite lo scheduler (con `published_at` impostato a una data futura) comparivano nel feed pubblico -- Homepage e pagine categoria -- ordinati per **data di creazione** (`created_at`), non per **data di pubblicazione effettiva** (`published_at`).

**Comportamento errato (esempio pratico):**

| Articolo | Creato il | Pubblicato il | Posizione attesa | Posizione reale |
|----------|-----------|---------------|-----------------|-----------------|
| Articolo A | 01 Mar | 01 Mar | 1 (piu recente) | 2 |
| Articolo B (schedulato) | 15 Feb | 22 Mar | 1 (piu recente) | 3 |
| Articolo C | 10 Feb | 10 Feb | 3 | 1 (errato) |

L'articolo B veniva creato il 15 Febbraio con pubblicazione schedulata al 22 Marzo. Quando si sbloccava, risultava ordinato dalla sua data di *creazione* (15 Feb), apparendo sepolto nel feed sotto articoli creati e pubblicati successivamente.

Il feed RSS (file `rss.php`) non aveva questo problema in quanto gia usava `ORDER BY published_at DESC`. La divergenza tra i due ordinamenti creava un'incongruenza tra il feed del sito e il feed RSS.

Questo bug era presente dalla v1.5.0 (introduzione dello scheduler).

### Causa Radice

La query principale di `articles.php` per il listing degli articoli utilizzava:
```sql
ORDER BY created_at DESC
```
invece di considerare `published_at` come criterio di ordinamento primario.

### Risoluzione

Sostituita la clausola `ORDER BY` con un'espressione `CASE` che usa `published_at` quando disponibile (non NULL e non stringa vuota, per compatibilita retroattiva con articoli legacy) e `created_at` come fallback:

```sql
ORDER BY CASE
    WHEN published_at IS NOT NULL AND published_at != ''
    THEN published_at
    ELSE created_at
END DESC
```

Questo garantisce:
- Gli articoli schedulati appaiono in cima al feed **al momento del loro sblocco**, nella posizione cronologica corretta rispetto agli altri articoli pubblicati.
- Gli articoli legacy (importati con `published_at` NULL o vuoto) continuano a ordinarsi per data di creazione, senza regressioni.
- Il comportamento del feed pubblico e' ora allineato con quello del feed RSS (`rss.php`).

**File modificato:** `public/api/articles.php` (riga 107)

---

## Note Tecniche

- Entrambe le fix riguardano esclusivamente `public/api/articles.php`.
- Nessuna modifica al database, nessuna migrazione necessaria.
- Nessuna modifica al frontend React.
- Nessun impatto sulle API admin (dashboard, editor): il parametro `?admin=true` con sessione valida continua a funzionare come prima.
