# Changelog v1.18.0 (29 Aprile 2026)

## Feature: Pin Articolo di Riferimento per Categoria

### Motivazione

Ogni categoria/sottocategoria del portfolio corrisponde di fatto a un progetto
(es. TelegramBot, SoundMaster Pro, ecc.). In questi casi è utile tenere fisso
in cima alla sezione un articolo specifico che descrive il progetto in modo
completo, indipendentemente dalla data di pubblicazione degli altri contenuti.

### Comportamento

- Nella tabella articoli del pannello admin compare una nuova colonna **📌**
  (pin di categoria), distinta dalla stella ★ (vetrina homepage).
- Cliccando il pin si fissa quell'articolo come **Articolo di Riferimento**
  per la sua categoria. Il backend rimuove automaticamente il pin da qualsiasi
  altro articolo della stessa categoria: può esistere **al massimo un pin per
  categoria**.
- Cliccando nuovamente si rimuove il pin.
- Nella pagina pubblica della categoria l'articolo pinned compare sempre
  **primo** (hero card grande), con badge **"Articolo di Riferimento"** al
  posto di "In Primo Piano".
- La vetrina homepage (★) e il pin di categoria (📌) sono indipendenti e
  possono coesistere sullo stesso articolo.

### Priorità di ordinamento (server + client)

```
is_category_pinned DESC  →  is_featured DESC  →  published_at DESC
```

### Modifiche tecniche

#### DB
- Nuova colonna `articles.is_category_pinned TINYINT(1) DEFAULT 0`
- Nuovo indice `idx_cat_pin (category, is_category_pinned)`
- Script di migrazione: `scripts/migrate_add_category_pin.php`

#### Backend (`public/api/articles.php`)
- GET: include `is_category_pinned` nel SELECT e nell'ORDER BY
- POST / PUT: persistono il campo `is_category_pinned`
- PATCH: nuovo ramo per `is_category_pinned`; quando si imposta a 1,
  azzera automaticamente il pin sugli altri articoli della stessa categoria

#### Frontend
- `src/api.ts`: aggiunto `toggleCategoryPin(id, bool)`
- `src/types.ts`: `PortfolioItem.isCategoryPinned`, `Article.is_category_pinned`
- `src/utils/mappers.ts`: mappatura `is_category_pinned → isCategoryPinned`
- `src/components/ArticleArchive.tsx`: ordinamento e badge aggiornati
- `src/pages/admin/ArticlesList.tsx`: colonna 📌 con `Pin`/`PinOff` icon

## Feature: Embed Video YouTube nell'Editor

### Comportamento

- Nella toolbar del `RichTextEditor` compare un pulsante YouTube (icona rossa).
- Click → si apre un input dove incollare l'URL del video
  (es. `https://www.youtube.com/watch?v=...`) o direttamente l'ID.
- Il video viene incorporato come iframe `youtube-nocookie.com` (privacy-safe),
  con layout responsive `aspect-video`, bordo e angoli arrotondati coerenti
  con il resto dell'editor.
- Supporto tasto `Enter` per confermare e `Escape` per chiudere.

### Modifiche tecniche

- Aggiunto `@tiptap/extension-youtube@^3.22.4`
- `src/components/admin/RichTextEditor.tsx`: import, estensione e UI toolbar
- `public/.htaccess`: CSP aggiornata con
  `frame-src https://www.youtube-nocookie.com https://www.youtube.com`

## Fix: Cache server-side (hard refresh mostra design vecchio)

### Problema

Su DreamHost, alcune URL di articoli venivano restituite con HTML stale
da un layer di cache server-side (OPcache / proxy Nginx), causando la
comparsa del vecchio design V2 al primo hard refresh su un browser vergine.

### Fix

- `public/index.php` (percorso umano): aggiunto
  `header('Cache-Control: no-store, no-cache, must-revalidate')`
  su ogni risposta, impedendo a qualsiasi layer intermedio di conservare l'HTML
  della SPA.
- `scripts/clear_cache.php`: script one-shot per svuotare OPcache e APCu
  residui sul server (da caricare via FTP, eseguire e cancellare subito).

### Note deploy

1. Caricare la `dist/` aggiornata sul server via FTP.
2. Eseguire `scripts/migrate_add_category_pin.php` sul server per aggiungere
   la colonna DB (FTP → esegui via browser → cancella).
3. Opzionale: eseguire `scripts/clear_cache.php` per forzare la pulizia
   della cache residua.

## Fix: Video YouTube non visibile nell'articolo pubblicato

### Problema

Il video YouTube inserito tramite l'editor appariva correttamente durante
la scrittura (Tiptap), ma scompariva nell'articolo pubblicato sul frontend.

### Causa

`DOMPurify.sanitize()` in `SingleArticle.tsx` rimuove tutti i tag `<iframe>`
per default, poiché considerati potenzialmente pericolosi.

### Fix — `src/components/SingleArticle.tsx`

Aggiunto hook `uponSanitizeElement` che consente il passaggio degli `<iframe>`
**esclusivamente** se il loro attributo `src` punta a:
- `https://www.youtube-nocookie.com/`
- `https://www.youtube.com/`

Qualsiasi altro iframe (potenziali vettori XSS) viene comunque rimosso.
Aggiunti anche gli attributi necessari al funzionamento dell'embed:
`allowfullscreen`, `frameborder`, `allow`, `src`.
