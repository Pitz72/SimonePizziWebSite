# Changelog v1.7.1 (04-04-2026)

## Hotfix Post-Migrazione: Ripristino Immagini e Compatibilità MySQL

Questa versione risolve tre bug critici emersi dopo la migrazione da SQLite a MySQL (v1.7.0), tutti scoperti e corretti in produzione.

---

## Bug 1 — HTTP 500 su articles.php (Blocco Totale del Sito)

### Sintomo
Dopo il deploy di v1.7.0, il sito mostrava "Errore recupero articoli" in homepage e in tutte le pagine di archivio. La console del browser riportava `articles.php: 500 Internal Server Error`. Il pannello admin era inaccessibile.

### Causa Radice
SQLite tratta tutti i valori come testo — il campo `published_at` era di tipo `TEXT` e poteva fisicamente contenere una stringa vuota `''`. Le query usavano il confronto:

```sql
-- SQLite: funzionava (campo TEXT)
WHERE published_at IS NULL OR published_at = '' OR published_at <= ?
ORDER BY CASE WHEN published_at IS NOT NULL AND published_at != '' THEN ...
```

MySQL ha un campo `DATETIME NULL` — confrontare un `DATETIME` con una stringa vuota `''` è una violazione di tipo che causa un errore fatale in **strict mode** (la modalità predefinita di MySQL 5.7+). PHP traduceva questo errore in HTTP 500.

### File Coinvolti
- `public/api/articles.php` (2 occorrenze)
- `public/api/rss.php` (1 occorrenza)

### Fix Applicato

**articles.php** — condizione WHERE (riga ~98):
```php
// PRIMA (SQLite)
$conditions[] = "(published_at IS NULL OR published_at = '' OR published_at <= ?)";

// DOPO (MySQL)
$conditions[] = "(published_at IS NULL OR published_at <= ?)";
```

**articles.php** — ORDER BY (riga ~115):
```php
// PRIMA (SQLite)
ORDER BY CASE WHEN published_at IS NOT NULL AND published_at != '' THEN published_at ELSE created_at END DESC

// DOPO (MySQL)
ORDER BY CASE WHEN published_at IS NOT NULL THEN published_at ELSE created_at END DESC
```

**rss.php** — condizione WHERE (riga ~34):
```php
// PRIMA (SQLite)
WHERE status = 'published' AND (published_at IS NULL OR published_at = '' OR published_at <= :ita_now)

// DOPO (MySQL)
WHERE status = 'published' AND (published_at IS NULL OR published_at <= :ita_now)
```

## Bug 2 — Immagini di Copertina Mancanti (Articoli e Progetti)

### Sintomo
22 articoli e tutti i 6 progetti mostravano l'immagine placeholder invece della copertina reale. La Media Gallery admin mostrava correttamente tutti i file. I file erano fisicamente presenti sul server in `/uploads/`.

### Causa Radice
Il bug ha due origini distinte che si sommavano:

**Origine A — Conversione WebP (introdotta in v1.6.3):**
Quando `upload.php` riceve un file immagine `.png` o `.jpg`, lo converte automaticamente in WebP e lo salva con estensione `.webp`. Questo comportamento è corretto. Tuttavia, i path già salvati nel database per gli articoli precedenti a v1.6.3 erano rimasti con le estensioni originali. Sul disco esisteva `/uploads/titan-icon.webp`; nel DB era salvato `/uploads/titan-icon.png`. Il frontend quindi cercava un file che non esisteva più.

Esempio:
```
DB:    /uploads/69aab0a465fa7-titan-icon.png   ← estensione vecchia
DISCO: /uploads/69aab0a465fa7-titan-icon.webp  ← file reale dopo conversione
```

**Origine B — URL Assoluti nel DB:**
4 record avevano il path salvato come URL assoluto completo (`https://simonepizzi.runtimeradio.it/uploads/...`) invece che come path relativo (`/uploads/...`). Questo era un residuo di inserimenti manuali o di una versione precedente dell'editor. In SQLite non causava problemi visivi perché il browser risolveva correttamente l'URL assoluto. In MySQL la normalizzazione ha reso più evidente l'inconsistenza.

### Fix Applicato
Script one-shot `fix_image_paths.php` eseguito direttamente in produzione:
1. Per ogni `cover_image` in `articles` e `projects` che conteneva `/uploads/`:
   - Normalizzava URL assoluti → path relativo (rimuovendo `https://simonepizzi.runtimeradio.it`)
   - Sostituiva l'estensione `.png`/`.jpg`/`.jpeg` con `.webp`
   - **Solo se il file `.webp` esisteva fisicamente sul disco** — nessuna modifica cieca
2. Aggiornava il record nel DB con il path corretto

## Nota Architetturale — Prevenzione Futura

Il problema degli URL assoluti nel DB è stato risolto. Per prevenire il problema delle estensioni:

- **`upload.php` (v1.6.3+):** quando viene caricata una nuova immagine, il path salvato nel DB è già `.webp` dal momento dell'upload — il problema non si ripresenta per le nuove immagini.
- **Immagini legacy (art.1–10):** usano path `/images/` con estensioni originali (`.png`, `.jpg`) e sono file statici non gestiti da `upload.php` — non vengono convertiti e non necessitano di fix.
