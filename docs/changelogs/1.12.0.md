### Riscrittura `index.php` ‚Äî Motore SEO v2.0

**Prima (v1.7.3):** `index.php` iniettava solo meta tag OG/Twitter nell'`<head>` di `index.html`. Il `<body>` restava `<div id="root"></div>` vuoto. I crawler dovevano eseguire JavaScript per vedere il contenuto.

**Ora (v2.0):** `index.php` rileva il tipo di visitatore tramite User-Agent:

| Visitatore | Comportamento |
|:---|:---|
| **Umano** (browser) | Inietta meta tag + canonical + JSON-LD ‚Üí serve la SPA React (comportamento identico a prima) |
| **Crawler** (Googlebot, Bingbot, ecc.) | Genera HTML completo: `<head>` con meta + `<body>` con contenuto reale (h1, article, testo, immagini, breadcrumb) |

**Tipi di pagina supportati:**
- **Homepage** ‚Üí h1, articoli recenti, progetti recenti
- **Articolo singolo** (`/categoria/slug`) ‚Üí header, excerpt, contenuto completo, breadcrumb
- **Categoria** (`/categoria`) ‚Üí h1, lista articoli con titolo e excerpt
- **Tutti i Progetti** ‚Üí lista progetti con nome e descrizione
- **Contatti** ‚Üí pagina statica con breadcrumb

**Dati strutturati JSON-LD:**
- Homepage ‚Üí `WebSite` + `Person`
- Articolo ‚Üí `Article` con `author`, `publisher`, `datePublished`, `mainEntityOfPage`
- Categoria ‚Üí `CollectionPage`
- Progetti ‚Üí `CollectionPage`
- Contatti ‚Üí `ContactPage`

**Canonical URL:** Aggiunto `<link rel="canonical">` su tutte le pagine per prevenire contenuti duplicati.

### Semplificazione `.htaccess`

**Prima:** Le regole per `robots.txt` e `sitemap.xml` usavano la condizione `RewriteCond %{REQUEST_FILENAME} !-f` (fallback solo se il file fisico non esiste). Questo causava il **bug v1.9.6** del 404 sporadico su `robots.txt`.

**Ora:** Le regole puntano **sempre** ai generatori PHP, senza condizioni sul filesystem:
```apache
RewriteRule ^sitemap\.xml$ sitemap.php [L,NC]
RewriteRule ^robots\.txt$ robots.php [L,NC]
```

Rimosse anche le condizioni ridondanti `!^/robots\.txt$` e `!^/sitemap\.xml$` dalla regola catch-all della SPA.
