# Changelog v1.5.8 — Security Hardening Sessioni PHP

**Data di Rilascio:** 23 Marzo 2026
**Tipo di Release:** Security Patch
**File Modificati:** `public/api/auth.php`

---

## Sommario

Questa release consolida la sicurezza del meccanismo di autenticazione del Mini-CMS attraverso tre hardening distinti sul ciclo di vita delle sessioni PHP. Le vulnerabilita corrette in questa versione non erano sfruttabili in isolamento in modo immediato, ma rappresentavano vettori di attacco reali in scenari composti (XSS + Session Hijacking, Session Fixation su reti condivise). Il codice precedente seguiva il pattern minimo funzionante; questa versione lo porta agli standard OWASP per la gestione delle sessioni web.

---

## Fix 1 — Cookie di Sessione: HttpOnly, Secure, SameSite=Strict

### Problema

Il cookie di sessione PHP (`PHPSESSID`) veniva emesso senza alcun flag di sicurezza. Questo esponeva l'utente admin a tre scenari di attacco:

- **HttpOnly assente**: un eventuale attacco XSS (anche in futuro, su componenti terze o input non sanitizzati) poteva leggere il cookie di sessione tramite `document.cookie` e trasmetterlo a un server avversario.
- **Secure assente**: il cookie veniva trasmesso anche su connessioni HTTP non cifrate (rilevante per accessi da reti WiFi pubbliche, proxy, o ambienti di test non HTTPS).
- **SameSite assente**: il cookie veniva inviato automaticamente su richieste cross-site, aprendo la porta ad attacchi CSRF (Cross-Site Request Forgery) dove una pagina malevola poteva triggerare azioni admin a nome dell'utente loggato.

### Causa

`session_start()` senza configurazione preliminare usa i default di PHP, che non includono questi flag.

### Risoluzione

Aggiunta configurazione sessione **prima** di `session_start()`:

```php
// PRIMA (v1.5.7 e precedenti)
session_start();

// DOPO (v1.5.8)
ini_set('session.cookie_httponly', 1);        // Cookie non leggibile da JavaScript
ini_set('session.cookie_secure', 1);         // Cookie solo su HTTPS
ini_set('session.cookie_samesite', 'Strict'); // Blocca invio cross-site
session_start();
```

**Nota tecnica:** `SameSite=Strict` e la policy piu restrittiva: il cookie non viene inviato nemmeno cliccando un link esterno verso il sito. Per un pannello admin privato questo e il comportamento corretto e desiderato.

## Fix 2 — Session Fixation Attack: session_regenerate_id()

### Problema

Dopo un login riuscito, il sistema manteneva lo stesso ID di sessione che l'utente aveva prima dell'autenticazione. Questo permetteva un attacco di tipo **Session Fixation**:

1. L'attaccante ottiene un ID di sessione valido (non autenticato) dal server — es. visitando la pagina di login
2. In qualche modo impone quell'ID alla vittima (URL manipulation, cookie injection su HTTP, LAN condivisa)
3. La vittima esegue il login con quell'ID
4. L'attaccante usa il medesimo ID, ora autenticato come admin

### Causa

Il blocco di login (v1.5.7) impostava correttamente `$_SESSION['user_id']` e `$_SESSION['username']`, ma non rigenerava l'ID di sessione, lasciando la sessione pre-autenticazione "promossa" ad admin.

### Risoluzione

```php
// PRIMA (v1.5.7 e precedenti)
if ($user && password_verify($password, $user['password_hash'])) {
    $_SESSION['user_id'] = $user['id'];
    $_SESSION['username'] = $user['username'];

// DOPO (v1.5.8)
if ($user && password_verify($password, $user['password_hash'])) {
    session_regenerate_id(true); // Genera nuovo ID, invalida il vecchio
    $_SESSION['user_id'] = $user['id'];
    $_SESSION['username'] = $user['username'];
```

Il parametro `true` in `session_regenerate_id(true)` elimina il file di sessione vecchio dal server, non solo generando un nuovo ID ma rendendo il vecchio ID completamente inutilizzabile.

## Fix 3 — Logout Incompleto: Invalidazione Corretta della Sessione

### Problema

Il logout precedente eseguiva solo `session_destroy()`. Questo aveva tre limiti:

1. **I dati `$_SESSION` rimanevano in memoria** per il resto dell'esecuzione dello script corrente
2. **Il cookie client non veniva invalidato**: il browser continuava a tenere il cookie `PHPSESSID` con il vecchio ID. Se PHP non aveva ancora cancellato il file di sessione server-side (race condition, cache), quel cookie poteva ancora funzionare per un breve periodo
3. **Non conforme al pattern OWASP** per la terminazione sicura delle sessioni

### Risoluzione

```php
// PRIMA (v1.5.7 e precedenti)
if (isset($data['action']) && $data['action'] === 'logout') {
    session_destroy();
    echo json_encode(['status' => 'success']);
    exit;
}

// DOPO (v1.5.8) — sequenza completa di logout sicuro
if (isset($data['action']) && $data['action'] === 'logout') {
    $_SESSION = [];  // 1. Svuota i dati di sessione in memoria
    if (ini_get('session.use_cookies')) {
        $params = session_get_cookie_params();
        setcookie(session_name(), '', time() - 42000,  // 2. Invalida il cookie nel browser
            $params['path'], $params['domain'],
            $params['secure'], $params['httponly']
        );
    }
    session_destroy();  // 3. Elimina il file di sessione dal server
    echo json_encode(['status' => 'success']);
    exit;
}
```

**Sequenza OWASP:** svuotare `$_SESSION` -> sovrascrivere il cookie con scadenza passata (time() - 42000) -> `session_destroy()`.

## Interventi Collaterali (Non Bug Fix)

### Rimozione fix_db.php dal repository

Lo script `public/api/fix_db.php` (one-shot per trim di slug/category, eseguito in produzione) e stato eliminato dal repository locale. Non aveva auth, non aveva piu utilita operativa e rappresentava una superficie d'attacco inutile (OWASP: rimozione funzionalita non necessarie).

### Spostamento init_db.php in /scripts/

Lo script `public/api/init_db.php` (inizializzazione schema DB) e stato spostato da `/public/api/` a `/scripts/`. Ragioni:
- Non deve mai essere raggiungibile via HTTP
- La sua natura e di utility di disaster recovery, non di endpoint API
- Rimarra nel repository come documentazione dello schema e strumento di ripristino, ma **non deve mai essere deployato**

## Impatto sul Funzionamento

- **Utenti finali (sito pubblico):** nessun impatto — le sessioni riguardano solo il pannello admin
- **Utente admin:** il login continua a funzionare identicamente. L'unica differenza osservabile e che dopo il login il cookie di sessione avra i flag di sicurezza visibili negli strumenti del browser (DevTools -> Application -> Cookies: HttpOnly, Secure, SameSite: Strict)
- **Sessioni esistenti:** dopo il deploy, le sessioni admin aperte prima dell'aggiornamento rimarranno valide fino alla loro naturale scadenza PHP (nessun forced logout)
