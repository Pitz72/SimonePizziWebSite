# Changelog v1.15.1 (27 Aprile 2026)

## 🔐 IronShield Security Patch — Follow-up: Criticità Medie e Lievi

Completamento del ciclo di hardening avviato con v1.15.0. Questa patch risolve le 10 criticità rimanenti (5 medie, 5 lievi) identificate nella code review sistematica del backend PHP e del frontend React.

---

### 🟡 Criticità Medie — Risolte

#### [SEC-07] Campo `status` articoli senza whitelist — `articles.php`

Qualsiasi stringa arbitraria veniva accettata come stato dell'articolo, potendo creare record in uno stato non gestito dalla logica di visibilità.

**Fix:** Whitelist esplicita — solo `'draft'` e `'published'` sono accettati. Qualsiasi altro valore viene silenziosamente normalizzato a `'draft'`. Applicato sia sul metodo POST che PUT.

---

#### [SEC-08] Link bottoni CTA non validati come URL — `articles.php`

I campi `button_a_link` e `button_b_link` accettavano qualsiasi stringa, inclusi URL con schema `javascript:` che sarebbero stati salvati nel DB e serviti nei tag `<a href>`.

**Fix:** Nuova funzione `sanitizeUrl()`: accetta solo URL con protocollo `http://`, `https://` o `mailto:`. Qualsiasi altro schema (incluso `javascript:`) viene rigettato restituendo stringa vuota. Validazione finale con `filter_var(FILTER_VALIDATE_URL)`. Aggiunto anche cap a 100 caratteri sulle label dei bottoni.

---

#### [SEC-09] Endpoint CTA click senza rate limiting — `analytics.php`

Il tracciamento dei click sui bottoni non aveva nessun limite, permettendo a bot di inflazionare illimitatamente le statistiche.

**Fix:** Rate limiting per-IP: max 10 click per articolo al minuto, identificati tramite `hash('sha256', IP . minuto_corrente)`. Risposta neutra al superamento del limite (non si rivela il blocco). Migrazione idempotente aggiunge le colonne `ip_hash` e `clicked_at` alla tabella `cta_clicks`.

---

#### [SEC-10] Path traversal latente in `media.php` prima di `unlink()`

La `file_path` recuperata dal DB veniva usata direttamente per costruire il percorso fisico del file da eliminare. Un valore malevolo nel DB avrebbe potuto cancellare file fuori dalla cartella uploads.

**Fix:** Doppia validazione prima di `unlink()`:
1. `str_starts_with($filePath, '/uploads/')` — blocca qualsiasi path che non inizi con la cartella attesa.
2. `realpath()` + confronto con `realpath(__DIR__ . '/../uploads')` — blocca path traversal con `../` o symlink.

---

#### [SEC-11] Reset password non invalidava le sessioni attive — `auth.php` + `auth_helper.php`

Dopo un reset della password, le sessioni PHP già aperte rimanevano valide. Un attaccante con una sessione attiva manteneva l'accesso anche dopo il cambio credenziali.

**Fix — meccanismo `session_version`:**
- Aggiunta colonna `session_version INT DEFAULT 0` alla tabella `users` (migrazione idempotente).
- Al **login**: `session_version` dell'utente viene salvata in `$_SESSION['session_version']`.
- Al **reset password**: `session_version` viene incrementata di 1 nel DB (`session_version + 1`).
- In **`Auth::check()`**: ad ogni request admin viene confrontata la `session_version` in sessione con quella nel DB. Se non corrispondono, la sessione viene distrutta e restituito HTTP 401. Questo espelle immediatamente qualsiasi sessione aperta prima del reset.

---

### 🟢 Criticità Lievi — Risolte

#### [SEC-12] Rollback ottimistico errato in `ReactionBar.tsx`

Il catch del rollback usava la variabile `counts` dalla closure, già aggiornata ottimisticamente, invece del valore originale pre-update.

**Fix:** Snapshot espliciti `prevCounts` e `prevMyReactions` catturati prima dell'aggiornamento ottimistico. Il catch ripristina esattamente questi snapshot, indipendentemente da aggiornamenti concorrenti.

---

#### [SEC-13] Ricerca LIKE senza cap sulla lunghezza — `articles.php`

Il parametro `$q` della ricerca full-text non aveva limite di lunghezza. Una stringa arbitrariamente lunga causava una query `LIKE %...%` molto lenta su colonne non indicizzate (`content`, `excerpt`).

**Fix:** `mb_substr($q, 0, 100)` — la query di ricerca è cappata a 100 caratteri prima di essere usata nella clausola LIKE.

---

#### [SEC-14] `application/octet-stream` tra i MIME accettati in upload — `upload.php`

Il tipo MIME `application/octet-stream` era nella whitelist come fallback per ZIP/RAR, ma copre qualsiasi binario — troppo permissivo come regola generale.

**Fix:** Rimosso `application/octet-stream` dalla whitelist. Sostituito con normalizzazione sicura: se `mime_content_type()` restituisce `octet-stream` **e** l'estensione è già stata validata come `zip` o `rar` (primo layer di controllo), il MIME viene normalizzato a `application/zip` prima del controllo whitelist. Doppio vincolo, nessuna apertura generica.

---

#### [SEC-15] Rate limiting login non considera proxy/CDN — `auth.php`

`$_SERVER['REMOTE_ADDR']` dietro un reverse proxy interno restituisce l'IP del proxy, non del client reale — tutti i tentativi di login venivano aggregati sotto lo stesso IP.

**Fix:** Nuova funzione `getClientIp()`:
- Se `REMOTE_ADDR` è un IP pubblico (connessione diretta), lo usa direttamente — non falsificabile.
- Se `REMOTE_ADDR` è privato/loopback (siamo dietro proxy interno), legge `X-Forwarded-For` e prende il primo IP pubblico valido.
- Questo previene sia lo spoofing (su connessioni dirette) sia il problema di aggregazione (su hosting con proxy).

---

#### [SEC-16] `published_at` accettava stringhe datetime non validate — `articles.php`

Il campo `published_at` veniva salvato senza verificare il formato. Una stringa malformata poteva causare comportamenti inattesi nelle query di scheduling e visibilità articoli.

**Fix:** `DateTime::createFromFormat('Y-m-d H:i:s', $published_at_raw)` — se la stringa non è un datetime valido nel formato atteso, viene usato `date('Y-m-d H:i:s')` come fallback sicuro. Applicato sia su POST che PUT.

---

### 🛠️ Note Tecniche

- **Migrazioni DB idempotenti:** `cta_clicks.ip_hash`, `cta_clicks.clicked_at`, `users.session_version` vengono aggiunte automaticamente alla prima richiesta — nessun intervento manuale sul DB.
- **Nessuna breaking change** per il frontend esistente.
- **`auth_helper.php`** ora effettua una query DB per ogni request admin (per verificare `session_version`). Su un sito monoutente con pochi endpoint admin, l'impatto è trascurabile.

### ⚠️ Azioni Richieste al Deploy

Ricaricare la `dist/` completa sul server più il `config.php` locale (contiene `BACKUP_CRON_SECRET`). Le colonne DB vengono aggiunte automaticamente.
