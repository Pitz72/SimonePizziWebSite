## Panoramica

La v1.6.1 corregge due fastidi operativi del backend admin: la mancanza delle date nella lista articoli e i nomi di file brutti al momento del download da parte dei visitatori.

---

## 1. Colonna "Data" nella Lista Articoli

### Problema
Nella lista admin degli articoli le colonne erano: Titolo, Categoria, Stato, Vetrina, Azioni.
Non era visibile nessuna informazione temporale, costringendo l'admin ad entrare in ogni articolo per conoscerne la data di pubblicazione o schedulazione.

### Soluzione
Aggiunta colonna **Data** tra Stato e Vetrina. Il comportamento è contestuale allo stato dell'articolo:

| Stato | Campo mostrato | Label | Colore |
|:---|:---|:---|:---|
| Bozza | `created_at` | Creato | grigio scuro |
| Pubblicato | `published_at` | Pubbl. | grigio chiaro |
| Programmato | `published_at` | Sched. | blu (coerente col badge stato) |

### File coinvolti
- `public/api/articles.php` — MODIFICATO: aggiunto `created_at` alla SELECT della lista admin (riga 89)
- `src/pages/admin/ArticlesList.tsx` — MODIFICATO: aggiunta colonna Data, helper `formatDate`, colSpan aggiornati a 6

### Nessuna migrazione DB richiesta
La colonna `created_at` esisteva già nel database (usata nell'ORDER BY). È stato sufficiente aggiungerla alla SELECT.

## 2. Download File con Nome Originale Pulito

### Problema
I file caricati nella Media Gallery venivano salvati su disco con un prefisso hash generato da `uniqid()`:
```
69c198a683d49-Il_Relitto_Silente_1.2.5.zip
```
Quando un visitatore del sito scaricava il file tramite link diretto `/uploads/...`, la finestra di download del browser mostrava il nome brutto prefissato.

### Causa
`upload.php` usa `uniqid() . '-' . $fileName` per evitare collisioni su disco. Il nome originale pulito era già salvato nella colonna `filename` del database fin dall'inizio, ma non veniva mai sfruttato per il download.

### Soluzione: Proxy PHP `download.php`

Creato `public/api/download.php`. Accetta `?id=123`, recupera dal DB il `filename` originale e il `file_path` reale, e serve il file con l'header:
```
Content-Disposition: attachment; filename="Il_Relitto_Silente_1.2.5.zip"
```

**Caratteristiche:**
- **Retrocompatibile**: i file già caricati funzionano subito perché il `filename` originale è salvato nel DB dalla v1.3.4
- **Sicuro**: nessun path traversal possibile — il percorso è sempre letto dal DB, mai dall'input utente
- **Pubblico**: nessuna auth richiesta (i link sono condivisi con i visitatori del sito)
- **Non invasivo**: i file su disco NON vengono rinominati, i link diretti `/uploads/...` già presenti negli articoli continuano a funzionare normalmente

### Aggiornamento Media Gallery

Il pulsante **"Copia Link"** e il pulsante **link** nell'overlay hover ora si comportano in modo differenziato:

| Tipo file | Copia Link | Pulsante link |
|:---|:---|:---|
| Immagine | URL diretto `/uploads/...` (per `<img src>`) | Apre in nuova scheda |
| File (ZIP, PDF, MP3, ecc.) | URL proxy `/api/download.php?id=X` | Scarica con nome originale |

### File coinvolti
- `public/api/download.php` — NUOVO
- `src/pages/admin/MediaGallery.tsx` — MODIFICATO: handleCopyUrl e link hover aggiornati

### Nessuna migrazione DB richiesta
La colonna `filename` con il nome originale esisteva già nel database dalla v1.3.4.
