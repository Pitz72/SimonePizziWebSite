# Changelog v1.15.0 (27 Aprile 2026)

## 🔐 IronShield Security Patch — Revisione e Hardening Completo del Backend PHP

Questa versione è dedicata interamente alla sicurezza. A seguito di una code review sistematica su tutto il backend PHP (endpoint API, SEO Engine, sistema di backup), sono state identificate e risolte **6 classi di vulnerabilità** classificate per gravità, per un totale di **23+ occorrenze** corrette su **17 file**. Non sono state introdotte nuove funzionalità.

---

### 🔴 Gravissima — Risolta

#### [SEC-01] Secret key hardcodata in `backup.php` (esposizione accesso cron al DB)

**Impatto:** Chiunque avesse accesso al repository poteva eseguire un dump completo del database chiamando `/api/backup.php?action=cron&secret=pizzi_backup_v1` senza autenticazione.

**Fix:**
- La costante `BACKUP_CRON_SECRET` è stata spostata in `config.php` (già gitignored) e generata con `bin2hex(random_bytes(32))` — 64 caratteri esadecimali casuali crittograficamente sicuri.
- Aggiornato `config.example.php` con il placeholder e le istruzioni di generazione.
- Il confronto è ora eseguito con `hash_equals()` (timing-safe) invece dell'operatore `!==`, prevenendo timing attacks sulla chiave.
- Se la costante non è definita in `config.php`, l'accesso è negato per default (fail-safe).

**File coinvolti:** `api/backup.php`, `api/config.php`, `api/config.example.php`

### 🟠 Gravi — Risolte

#### [SEC-02] Debug attivo in produzione in `backup.php`

**Impatto:** `error_reporting(E_ALL)` e un `error_log` puntato a `api/.data/php_error.log` (dentro la webroot) erano attivi in produzione con commento "verrà rimosso dopo il fix". Il log PHP — potenzialmente accessibile via HTTP — poteva esporre path assolute del server, nomi di tabelle e stack trace.

**Fix:** Rimosse integralmente le tre righe `ini_set('log_errors')`, `ini_set('error_log')` e `error_reporting(E_ALL)`.

**File coinvolti:** `api/backup.php`

---

#### [SEC-03] File ZIP temporanei scritti nella webroot

**Impatto:** Gli archivi ZIP generati dal Full Content Exporter venivano salvati in `api/.data/temp/`, una sottocartella dentro `public/api/`. Se Apache non proteggeva esplicitamente questa directory, i file erano scaricabili da chiunque indovinasse il nome (pattern predicibile `export_pizzi_YYYY-MM-DD_HH-II-SS.zip`). In caso di interruzione della connessione, il file rimaneva esposto indefinitamente.

**Fix:** La temp dir è stata spostata in `sys_get_temp_dir() . '/pizzi_export'` — fuori dalla document root, inaccessibile via HTTP. I permessi della directory sono stati abbassati a `0700` (solo il processo PHP può accedervi). Il `@unlink()` post-download rimane invariato come cleanup.

**File coinvolti:** `api/backup.php`

---

#### [SEC-04] Messaggi di errore PDO/Exception esposti nelle risposte JSON

**Impatto:** Tutti gli endpoint PHP restituivano il messaggio grezzo di `$e->getMessage()` direttamente nelle risposte JSON in caso di errore. I messaggi PDO contengono tipicamente: nome della tabella, struttura della query, path del server, DSN di connessione. Un attaccante poteva provocare deliberatamente errori per fare information gathering sulla struttura del database.

**Fix:** In tutti i file coinvolti, la pattern `echo json_encode(['error' => $e->getMessage()])` è stata sostituita con:
1. `error_log(basename(__FILE__, '.php') . ' error: ' . $e->getMessage())` — l'errore reale viene scritto nel log di sistema del server.
2. `echo json_encode(['error' => 'Errore interno del server.'])` — l'utente riceve solo un messaggio generico.

Sono stati inoltre corretti gli artefatti di doppia concatenazione (`'Errore server: ' . 'Errore interno del server.'`) lasciati da una precedente patch parziale in `messages.php`, `newsletter_send.php` e `subscribers.php`.

**File coinvolti (23 occorrenze totali):**
`api/auth.php`, `api/articles.php`, `api/analytics.php`, `api/media.php`, `api/backup.php`, `api/categories.php`, `api/db.php`, `api/messages.php`, `api/navigation.php`, `api/newsletter_send.php`, `api/projects.php`, `api/search.php`, `api/settings.php`, `api/stats.php`, `api/subscribers.php`, `api/tags.php`, `api/upload.php`

#### [SEC-05] XSS latente nel SEO Engine — contenuto articolo iniettato raw per i crawler

**Impatto:** In `public/index.php`, il percorso di rendering per i crawler iniettava il campo `content` dell'articolo direttamente nell'HTML senza sanitizzazione (`'<div>' . $article['content'] . '</div>'`). Un contenuto malevolo nel campo (es. inserito bypassando l'editor o tramite un'eventuale vulnerabilità futura) avrebbe potuto rompere la struttura HTML della pagina servita ai crawler e iniettare script in contesti non attesi (chat app con preview, tool SEO con rendering JS).

**Fix:** Il contenuto è ora filtrato con `strip_tags()` con whitelist esplicita dei tag semantici ammessi:
```
<p> <br> <h2> <h3> <h4> <ul> <ol> <li> <strong> <em> <a> <blockquote> <pre> <code>
```
Gli attributi dei tag vengono rimossi automaticamente da `strip_tags()`, mantenendo solo il testo strutturato necessario per i crawler. La SPA React (percorso umano) non è interessata da questa modifica.

**File coinvolti:** `public/index.php`

---

#### [SEC-06] Rate limiting form contatti globale — attacco DoS sul form

**Impatto:** Il rate limiting in `messages.php` contava i messaggi **globali** degli ultimi 15 minuti, non per singolo IP. Un attaccante poteva inviare 10 messaggi consecutivi e bloccare il form di contatto per **tutti** i visitatori del sito per 15 minuti — un denial of service banale e silenzioso.

**Fix:**
- Il rate limiting è ora **per-IP**: vengono contati solo i messaggi con lo stesso `ip_hash` (SHA-256 dell'IP del mittente) negli ultimi 15 minuti.
- Soglia abbassata a **3 messaggi per IP** in 15 minuti (più coerente con un uso legittimo del form).
- Aggiunta la colonna `ip_hash VARCHAR(64)` alla tabella `messages` con migrazione idempotente (`ALTER TABLE ... ADD COLUMN IF NOT EXISTS`) che non rompe installazioni esistenti.
- Il campo `ip_hash` viene ora popolato a ogni salvataggio del messaggio.

**File coinvolti:** `api/messages.php`

---

### 🛠️ Note Tecniche

- Nessuna modifica al frontend React/TypeScript.
- Nessuna nuova dipendenza npm o PHP.
- La migrazione del DB (`ALTER TABLE messages ADD COLUMN ip_hash`) è idempotente: se la colonna esiste già, il catch interno la ignora silenziosamente.
- Il file `config.php` (gitignored) deve essere aggiornato manualmente sul server con la nuova costante `BACKUP_CRON_SECRET` prima del deploy.

### ⚠️ Azioni Richieste al Deploy

1. **Caricare `config.php`** aggiornato sul server via FTP (contiene `BACKUP_CRON_SECRET`).
2. **Caricare tutti i file PHP modificati** nelle rispettive cartelle sul server.
3. Verificare che Apache abbia i permessi per scrivere in `sys_get_temp_dir()` (di norma `/tmp` su Linux — già accessibile al processo PHP).
4. Il vecchio `api/.data/temp/` può essere rimosso manualmente dal server se presente.
