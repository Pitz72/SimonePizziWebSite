# Changelog v1.5.9 — Security Fix: Password Casuale init_db & Deprecation FILTER_SANITIZE_STRING

**Data di Rilascio:** 23 Marzo 2026
**Tipo di Release:** Security Patch
**File Modificati:** `scripts/init_db.php`, `public/index.php`

---

## Sommario

Questa release risolve due problemi di sicurezza distinti: la password amministrativa hardcoded nello script di inizializzazione del database e l'uso di una funzione PHP deprecata (rimossa in PHP 9) nel router SEO server-side. Entrambi i fix sono a basso impatto funzionale ma ad alto valore di sicurezza e manutenibilita a lungo termine.

---

## Fix 1 — init_db.php: Password Hardcoded Eliminata

### Problema

Lo script `scripts/init_db.php` (strumento di disaster recovery per la creazione del database da zero) conteneva due vulnerabilita distinte nel blocco di creazione dell'utente admin:

**1a — Password hardcoded in chiaro:**
```php
// PRIMA (v1.5.8 e precedenti)
$hash = password_hash('admin2026', PASSWORD_DEFAULT);
$pdo->exec("INSERT INTO users (username, password_hash) VALUES ('admin', '$hash')");
$logs[] = "Utente 'admin' creato (password iniziale: admin2026).";
```

La stringa `admin2026` era visibile:
- Nel codice sorgente del repository (chiunque acceda al repo conosce la password di ripristino)
- Nel log di output dello script, mostrato in chiaro via browser
- Come commento nella riga precedente

**1b — String interpolation nell'INSERT SQL:**
Anche se `$hash` e un hash bcrypt (caratteri sicuri per SQL), usare la concatenazione di stringa nell'`exec()` e una pratica scorretta: introduce un pattern pericoloso che, se replicato con variabili non sanitizzate, aprirebbe a SQL injection.

### Contesto Importante

La password di produzione sul server web era gia stata cambiata dall'admin tramite il pannello `/admin/settings` prima di questa release. Il rischio residuo di questa vulnerabilita era quindi limitato agli scenari di disaster recovery futuri (se qualcuno avesse dovuto eseguire `init_db.php` su un nuovo server).

### Risoluzione

```php
// DOPO (v1.5.9)
$tempPassword = bin2hex(random_bytes(12)); // 24 caratteri hex casuali, crittograficamente sicuri
$hash = password_hash($tempPassword, PASSWORD_DEFAULT);
$stmtInsert = $pdo->prepare("INSERT INTO users (username, password_hash) VALUES (?, ?)");
$stmtInsert->execute(['admin', $hash]);
$logs[] = "Utente 'admin' creato. PASSWORD TEMPORANEA: " . $tempPassword . " — CAMBIARLA IMMEDIATAMENTE dal pannello /admin/settings dopo il primo accesso. Questa password non viene salvata da nessuna parte.";
```

**Comportamento nuovo:**
- `bin2hex(random_bytes(12))` genera 12 byte casuali crittograficamente sicuri (via CSPRNG di PHP), convertiti in 24 caratteri esadecimali
- La password e unica per ogni esecuzione, non predicibile, non documentata nel codice
- Viene mostrata **una sola volta** nell'output JSON al momento dell'esecuzione — l'admin deve annotarla subito
- Aggiunto blocco `else` per gestire il caso in cui la tabella `users` sia gia popolata (nessuna sovrascrittura dei dati esistenti)
- L'INSERT usa un prepared statement PDO con parametri `?` (nessuna interpolazione di stringhe)

**Nota operativa per disaster recovery:** quando si esegue `init_db.php` su un server vergine, l'output JSON conterra il campo `logs` con la password temporanea. Annotarla, accedere al pannello admin, cambiarla immediatamente.

## Fix 2 — index.php: Rimozione FILTER_SANITIZE_STRING Deprecata

### Problema

Il router SEO server-side (`public/index.php`) usava `FILTER_SANITIZE_STRING` per sanitizzare lo slug estratto dall'URL prima di passarlo a una query PDO:

```php
// PRIMA (v1.5.8 e precedenti)
$cleanSlug = filter_var($slug, FILTER_SANITIZE_STRING);
```

**Perche e un problema:**
- `FILTER_SANITIZE_STRING` e stata **deprecata in PHP 8.1** (agosto 2021) e rimossa in PHP 9
- Il server usa PHP 8.x: da PHP 8.1 emette un `E_DEPRECATED` nel log ad ogni richiesta HTTP, appesantendo i log
- In vista di un eventuale upgrade a PHP 9, questo avrebbe causato un errore fatale bloccando il sito
- La funzione aveva un comportamento inconsistente tra versioni PHP (strip di tag HTML con side effect su caratteri speciali)

### Perche Non Era un Rischio SQL Immediato

Lo slug passava comunque a un PDO prepared statement (`execute([':slug' => $cleanSlug])`), che previene SQL injection indipendentemente dalla sanitizzazione precedente. La criticita era quindi principalmente di **compatibilita futura** e **pulizia dei log**, non di sicurezza immediata.

### Risoluzione

```php
// DOPO (v1.5.9)
$cleanSlug = strip_tags(trim($slug));
```

**Perche strip_tags + trim:**
- `strip_tags()` rimuove eventuali tag HTML/XML dalla stringa — equivalente funzionale al comportamento principale di `FILTER_SANITIZE_STRING`
- `trim()` rimuove spazi e caratteri di controllo iniziali/finali
- Nessuna deprecazione: entrambe le funzioni sono stabili in tutte le versioni PHP attive e future
- Il PDO prepared statement a valle garantisce comunque la protezione da SQL injection

## Impatto sul Funzionamento

- **Sito pubblico:** nessun cambiamento visibile. La modifica a `index.php` e interna al processo di sanitizzazione dello slug — il comportamento SEO e identico.
- **Utente admin attivo:** nessun impatto. La sessione, la password e il pannello admin rimangono invariati.
- **Disaster recovery futuro:** se si dovesse mai eseguire `init_db.php` su un nuovo server, la password admin non sara piu `admin2026` ma una stringa casuale mostrata nell'output. Annotarla prima di chiudere la pagina.

## Note Tecniche

### bin2hex(random_bytes(12)) vs password_hash() con salt manuale

`random_bytes()` in PHP usa il CSPRNG del sistema operativo (`/dev/urandom` su Linux, `CryptGenRandom` su Windows). L'output e crittograficamente sicuro e non predicibile. La conversione con `bin2hex()` produce una stringa di soli caratteri alfanumerici (0-9, a-f) — facile da annotare, impossibile da indovinare, non contiene caratteri speciali che potrebbero causare problemi in form HTML.

### Comportamento FILTER_SANITIZE_STRING vs strip_tags

| Comportamento | FILTER_SANITIZE_STRING | strip_tags(trim()) |
| :--- | :--- | :--- |
| Rimozione tag HTML | Si | Si |
| Trim whitespace | No (solo interno) | Si (esplicito con trim()) |
| Encoding entita HTML | Si (side effect) | No |
| Stato PHP 8.1+ | DEPRECATED | Stabile |
| Stato PHP 9 | RIMOSSA | Stabile |

Per un URL slug destinato a query PDO, `strip_tags(trim())` e la sostituzione piu corretta e piu prevedibile.
