## Fix 1 — DOMPurify: Protezione XSS Stored in SingleArticle (MEDIA)

### Problema

`SingleArticle.tsx` iniettava il contenuto HTML degli articoli direttamente nel DOM tramite `dangerouslySetInnerHTML` senza alcuna sanitizzazione lato frontend:

```tsx
// PRIMA (v1.5.9 e precedenti)
dangerouslySetInnerHTML={{ __html: article.description }}
```

Se un contenuto malevolo fosse stato inserito nel campo `description` di un articolo (tramite accesso admin compromesso o vulnerabilita future), sarebbe stato eseguito nel browser di ogni visitatore (XSS Stored).

### Risoluzione

Installata la libreria `dompurify` (standard de-facto per la sanitizzazione HTML nel browser):

```bash
npm install dompurify
npm install --save-dev @types/dompurify
```

```tsx
// DOPO (v1.5.10)
import DOMPurify from 'dompurify';

dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(article.description) }}
```

DOMPurify analizza il DOM in un contesto sandbox, rimuove tutti gli elementi e attributi pericolosi (`<script>`, `onerror=`, `javascript:`, `data:text/html`, ecc.) e restituisce HTML sicuro. Il contenuto editoriale legittimo (grassetto, link, immagini, blockquote, codice) passa inalterato.

**Nota:** contestualmente `npm audit fix` ha aggiornato rollup da una versione con vulnerabilita path traversal (GHSA-mw96-cpmx-2vgc) alla versione corrente. 0 vulnerabilita note al momento del rilascio.

## Fix 2 — Security Headers HTTP via .htaccess (MEDIA)

### Problema

Il server non inviava alcun header di sicurezza HTTP. Questo lasciava il sito esposto a:
- **MIME-type sniffing**: il browser poteva eseguire un file .jpg come script se il server inviava il MIME errato
- **Clickjacking**: il sito poteva essere caricato in un iframe di una pagina malevola per ingannare i click degli utenti
- **Referrer leakage**: URL interne (incluse quelle del pannello admin) potevano essere inviate come Referer a siti terzi
- **API hardware non necessarie**: accesso a camera, microfono, GPS aperto per default

### Risoluzione

Aggiunto blocco `<IfModule mod_headers.c>` in `public/.htaccess`:

```apache
Header always set X-Content-Type-Options "nosniff"
Header always set X-Frame-Options "SAMEORIGIN"
Header always set Referrer-Policy "strict-origin-when-cross-origin"
Header always set Permissions-Policy "camera=(), microphone=(), geolocation=()"
Header always set Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https: blob:; connect-src 'self'; font-src 'self' data:; object-src 'none'; frame-ancestors 'self'; base-uri 'self'"
```

**Dettaglio CSP:**
| Direttiva | Valore | Motivazione |
| :--- | :--- | :--- |
| `script-src` | `'self'` | Solo bundle Vite locali; nessun inline script in produzione |
| `style-src` | `'self' 'unsafe-inline'` | CSS Vite + `style=""` inline usati da Framer Motion per le animazioni |
| `img-src` | `'self' data: https: blob:` | Immagini locali + data URI + copertine articoli da CDN esterni |
| `connect-src` | `'self'` | Chiamate API React solo verso stesso dominio |
| `object-src` | `'none'` | Blocca plugin Flash/PDF embed (vettore attacco legacy) |
| `frame-ancestors` | `'self'` | Equivale a X-Frame-Options SAMEORIGIN per browser moderni |

## Fix 3 — upload.php: mkdir 0777 → 0755 (MEDIA)

### Problema

La creazione della directory `/uploads/` usava permessi `0777` (rwxrwxrwx): lettura, scrittura ed esecuzione per tutti gli utenti di sistema.

```php
// PRIMA
mkdir($uploadDir, 0777, true);
```

Su server condivisi o con utenti multipli, `0777` permetteva a qualsiasi processo del sistema di scrivere nella directory degli upload, potenzialmente sovrascrivendo file o iniettando contenuto.

### Risoluzione

```php
// DOPO
mkdir($uploadDir, 0755, true); // rwxr-xr-x: owner pieno, altri solo lettura/esecuzione
```

`0755` e il permesso standard per directory web: il processo PHP (owner) ha pieno controllo, gli altri utenti possono solo listare/leggere.

## Fix 4 — generateSlug: Normalizzazione Accenti Italiani (LIEVE)

### Problema

La funzione `generateSlug()` in `articles.php` usava `preg_replace('/[^A-Za-z0-9-]+/', '-', $title)` che rimuoveva i caratteri accentati italiani senza sostituirli, producendo slug errati:

- "Caffe con carattere" → `caffe-con-carattere` (corretto)
- "Il caffe e pronto" → `il-caff--pronto` (SBAGLIATO: doppio trattino)
- "Riunione all'alba" → `riunione-all-alba` (casuale)
- "Un po' di tutto" → `un-po--di-tutto` (SBAGLIATO)

### Risoluzione

Aggiunta translitterazione esplicita prima del `preg_replace`:

```php
$accents      = ['a','e','e','i','o','u','a','e','e','i','o','u','a','e','i','o','u','a','e','i','o','u'];
$replacements = ['a','e','e','i','o','u','a','e','e','i','o','u','a','e','i','o','u','a','e','i','o','u'];
$title = str_replace($accents, $replacements, $title);
```

Copre: `a e e i o u` (minuscolo e maiuscolo) piu caratteri francesi (`a e i o u` con accento circonflesso) e tedeschi (`a e i o u` con umlaut).

## Fix 5 — execCommand Deprecated in RichTextEditor (LIEVE)

### Problema

`document.execCommand()` e deprecated nelle specifiche Web e genera warning nei DevTools dei browser moderni. Usato per tutte le operazioni dell'editor WYSIWYG: grassetto, corsivo, liste, allineamento, colori, inserimento HTML da paste.

### Soluzione Adottata (Pragmatica)

La migrazione completa a Selection/Range API richiederebbe di riscrivere ogni comando dell'editor (bold, italic, lists, foreColor, formatBlock, ecc.) — equivalente a costruire un editor da zero. Poiche:
- `execCommand` e ancora supportato da tutti i browser nel 2026 (Chrome, Firefox, Safari, Edge)
- Chromium non ha annunciato date di rimozione imminenti
- La criticita e **LIEVE**

Il fix adottato in questa versione e un hardening difensivo:

```tsx
// PRIMA
const exec = (command: string, value?: string) => {
    document.execCommand(command, false, value);
    ...
};

// DOPO
const exec = (command: string, value?: string) => {
    try {
        document.execCommand(command, false, value); // eslint-disable-line
    } catch {
        console.warn(`[RichTextEditor] execCommand '${command}' non supportato.`);
    }
    ...
};
```

Stesso pattern applicato ai due `execCommand('insertHTML')` nel paste handler.

**Roadmap:** la migrazione completa a Selection/Range API (o adozione di TipTap/ProseMirror) e tracciata come task futuro.

## Fix 6 — RSS LIMIT 50: Costante Nominata (LIEVE)

### Problema

Il valore `LIMIT 50` nella query del feed RSS era un magic number hardcoded inline nella stringa SQL, difficile da trovare e modificare.

### Risoluzione

```php
// PRIMA
LIMIT 50

// DOPO
define('RSS_FEED_LIMIT', 50);
// ...
LIMIT " . RSS_FEED_LIMIT
```

Il valore rimane 50 ma ora e modificabile da un unico punto con nome semantico.

## Fix 7 — Migrazione DB: Colonna updated_at (MEDIA — Pendente Deploy)

### Problema

La tabella `articles` non aveva una colonna `updated_at` per tracciare l'ultima modifica di un articolo. Questo impediva:
- Ordinamenti per "ultima modifica"
- Cache invalidation basata su data modifica
- Metadati SEO `<meta name="last-modified">`

### Risoluzione

Creato `scripts/migrate_add_updated_at.php` — script di migrazione idempotente (sicuro da rieseguire) che:
1. Verifica con `PRAGMA table_info(articles)` se la colonna esiste gia
2. Se assente, esegue `ALTER TABLE articles ADD COLUMN updated_at DATETIME DEFAULT CURRENT_TIMESTAMP`
3. Popola `updated_at` per gli articoli esistenti con `COALESCE(published_at, created_at)`

**Per applicare in produzione:**
1. Carica `scripts/migrate_add_updated_at.php` via FTP in `/public/api/` sul server
2. Visita `https://simonepizzi.it/api/migrate_add_updated_at.php`
3. Verifica output JSON: `"status": "success"`
4. Cancella il file dal server FTP immediatamente
