# Changelog v1.5.5 - Sync Timezone Server: Los Angeles -> Roma

**Data Rilascio:** 06 Marzo 2026

## 🎯 Obiettivo della Release
Questa patch vitale corregge un bug logico nell'architettura nativa del file-system/database distribuito sull'hosting dell'utente. Il server web si trova fisicamente nella regione di Los Angeles (PST / UTC-8 o PDT / UTC-7), causando un ritardo costante e tassativo di ben 9 ore nello sblocco "programmatico" dei nuovi articoli gestiti tramite l'editor temporale del CMS.

## ✨ Dettaglio Modifiche

### 1. Bypass Ora di Sistema in API ed RSS
- **Problema Architetturale**: L'API PHP che estrae gli articoli per disegnarli sul frontend (`articles.php`) e lo script che genera il podcast per le piattaforme esterne (`rss.php`) domandavano al database SQLite di validare la riga tramite la formula matematica interna `datetime('now', 'localtime')`. Purtroppo il localhost per un server Californiano si materializza sempre 9 ore dopo rispetto all'Europa. Se un admin italiano fissa l'uscita per "oggi alle 09:00", il server nasconderà il materiale fino alle ore 18:00 italiane.
- **Risoluzione Applicata**: Entrambi i file core (`articles.php` e `rss.php`) ora implementano in *header* la direttiva incondizionata PHP `date_default_timezone_set('Europe/Rome');`. Il costrutto di SQLite è stato conseguentemente sostituito tramite Binding. Piuttosto che far ragionare il motore SQL in proprio, ora PHP genera una stringa "Adesso" tarata perfettamente sui cronometri italiani, inviando a SQLite una richiesta statica ed assoluta (Es. `published_at <= '2026-03-06 09:00:00'`).

### 2. Impatti Collaterali (Effetto Retroattivo)
L'aggiornamento ha applicato in tempo zero la sincronizzazione temporale e ha forzato lo sblocco per l'utente e per il Feed di qualsiasi post che giaceva erroneamente tra lo scarto delle presunte "18:00 italiane" (Le 9 di mattina di L.A.).
